# System Design

## Purpose

Cicosy CRM is a multi-tenant CRM platform with sales, service, marketing, reporting, field-service, integrations, realtime updates, document storage, and AI-assisted workflows.

## Runtime Components

- React frontend: user workspace, CRM modules, AI chat, reporting, settings, onboarding, realtime subscriptions.
- Java backend: system of record for CRM data, auth, RBAC, tenant routing, reporting, integrations, documents, notifications, and realtime events.
- Python AI service: conversational assistant, RAG, AI insights, AI governance, audit, safe action proposals, and AI observability.
- PostgreSQL: primary relational store with tenant-aware schema and Flyway migrations.
- Redis: cache layer.
- RabbitMQ: asynchronous event backbone where enabled by deployment.
- Object storage: MinIO/S3-compatible document storage.
- Observability stack: Spring Actuator, AI `/metrics`, Prometheus, Grafana starter assets.

## High-Level Flow

1. User authenticates against the Java backend.
2. Backend issues JWT access and refresh tokens with tenant, user, and session context.
3. Frontend calls Java backend for CRM data and Python AI service for AI interactions.
4. AI service uses the user's JWT when it needs CRM data, so Java RBAC and tenant isolation remain authoritative.
5. Backend publishes realtime events for changed entities.
6. Documents are stored in object storage and referenced by metadata in PostgreSQL.
7. Standard CRM reports are generated by Java; custom AI reports are generated by Python.

## Tenancy Model

- Every tenant-owned record has `tenant_id`.
- Java request processing sets tenant context from the authenticated JWT.
- Hibernate tenant filters and explicit repository filters enforce application-level isolation.
- Async execution preserves tenant/security context so background work cannot silently detach from the active workspace.
- Tenant-owned cache entries are keyed by tenant plus record identifier.
- Tenant-owned queue messages must carry tenant identifiers and fail closed when the identifier is missing.
- Database RLS policies exist as defense-in-depth and are being expanded for newer tables.
- Dedicated tenant database routing is supported for enterprise tenants, but should remain an admin-only operational concept.

## AI Boundary

The AI service is not the source of truth. It can:

- Read CRM context through the Java backend.
- Answer questions using authenticated CRM tools and RAG.
- Generate insights and recommendations.
- Propose safe writeback actions.
- Execute only explicitly confirmed low/medium-risk actions.
- Route high-risk actions to approval workflows.

The AI service must not bypass Java authorization or tenant isolation.
